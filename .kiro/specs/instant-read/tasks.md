# Implementation Plan

## Tasks

- [x] 1. プロジェクト初期セットアップ
- [x] 1.1 Tauri v2 + SolidJS プロジェクトをスキャフォルドし、開発環境を構築する
  - `pnpm create tauri-app` で SolidJS + TypeScript テンプレートを生成
  - Tailwind CSS を導入しダークテーマをデフォルトに設定
  - Tauri プラグイン（cli, single-instance, window-state）をインストール
  - ウィンドウ設定で非表示起動（hidden window pattern）を有効化
  - `pnpm tauri dev` でアプリが起動することを確認
  - _Requirements: 2.2, 5.3, 6.1_

- [x] 1.2 CLI引数の定義と受け渡しを設定する
  - tauri.conf.json にファイルパス引数とディレクトリ引数を定義
  - Rust 側で CLI 引数を解析しフロントエンドに渡す仕組みを構築
  - capabilities に cli プラグインの権限を追加
  - _Requirements: 1.1, 1.2, 1.3_

- [x] 2. Rust バックエンド IPC コマンド
- [x] 2.1 (P) ファイル読み取りコマンドを実装する
  - 指定パスのファイル内容をテキストとして読み取り返すコマンドを作成
  - ファイル不存在時にパスを含む構造化エラーを返す
  - パストラバーサル防止のためパスを正規化する
  - Tauri の fs 権限スコープを設定し読み取りアクセスを制限
  - Rust ユニットテストで正常系・異常系を検証
  - _Requirements: 1.1, 1.4, 1.5_

- [x] 2.2 (P) ディレクトリ内 Markdown ファイル一覧コマンドを実装する
  - 指定ディレクトリを走査し .md/.markdown ファイルの一覧を返すコマンドを作成
  - 各エントリにファイル名、パス、更新日時、サイズを含める
  - ディレクトリ不存在・空ディレクトリ時のエラーハンドリング
  - Rust ユニットテストで正常系・空ディレクトリ・不存在を検証
  - _Requirements: 1.2, 1.3_

- [ ] 3. Markdown レンダリングパイプライン
- [ ] 3.1 unified/remark/rehype パイプラインを構築する
  - unified, remark-parse, remark-gfm, remark-rehype, rehype-stringify をインストール
  - Markdown 文字列を HTML 文字列に変換するサービスモジュールを作成
  - GFM 拡張（テーブル、チェックリスト、取り消し線、脚注）の変換を検証
  - 基本 Markdown 要素（見出し、リスト、引用、リンク、画像）の変換を検証
  - _Requirements: 5.1, 5.4_

- [ ] 3.2 シンタックスハイライトを統合する
  - rehype-pretty-code と shiki をインストール
  - github-dark テーマでコードブロックのハイライトを有効化
  - 主要言語（JavaScript, TypeScript, Rust, Python, JSON, YAML, Bash）で表示を検証
  - _Requirements: 5.2_

- [ ] 3.3 HTML サニタイズを追加し安全なレンダリングを実現する
  - rehype-sanitize をパイプラインに追加
  - 危険な HTML タグやスクリプトが除去されることを検証
  - 外部リンクに target="_blank" と rel="noopener noreferrer" を自動付与
  - _Requirements: 5.1, 5.4_

- [ ] 4. プレビュー UI コンポーネント
- [ ] 4.1 (P) プレビューコンポーネントを作成し美しいタイポグラフィを適用する
  - HTML 文字列を受け取りレンダリングする SolidJS コンポーネントを作成
  - ダークテーマの背景色、テキスト色を Tailwind で設定
  - 行間、フォントサイズ、マージン、最大幅を読みやすさに最適化
  - GFM テーブル、チェックリスト、コードブロックのスタイリング
  - スクロール可能なコンテナとして構成
  - _Requirements: 5.3, 5.5_

- [ ] 4.2 (P) エラー表示コンポーネントを作成する
  - ファイルパスを含むエラーメッセージを見やすく表示するコンポーネント
  - エラー種別（ファイル不存在、権限不足、パースエラー）ごとのメッセージ
  - ダークテーマに馴染むデザイン
  - _Requirements: 1.4_

- [ ] 4.3 (P) ファイル一覧コンポーネントを作成する
  - ディレクトリ内の Markdown ファイルを一覧表示するコンポーネント
  - ファイル名、更新日時、サイズを表示
  - ファイルをクリックするとプレビュー表示に切り替わる
  - ダークテーマに馴染むデザイン
  - _Requirements: 1.2, 1.3_

- [ ] 5. アプリケーションシェルと状態管理
- [ ] 5.1 App ルートコンポーネントで表示状態を管理する
  - 現在のファイルパスと表示モード（プレビュー/ファイル一覧/エラー）を Signal で管理
  - ファイル内容を createResource で IPC read_file コマンドと連動させる
  - Markdown パイプラインの結果を Preview コンポーネントに渡す
  - 非 Markdown ファイルをプレーンテキストとして表示する分岐
  - _Requirements: 1.1, 1.5, 6.1_

- [ ] 5.2 CLI 引数からの初期表示を接続する
  - Rust から渡された CLI 引数（ファイルパス / ディレクトリパス）を受け取る
  - ファイル指定時はプレビュー表示、ディレクトリ指定時はファイル一覧、引数なしはカレントディレクトリの一覧を表示
  - Hidden window を表示に切り替えるタイミング制御（レンダリング完了後）
  - _Requirements: 1.1, 1.2, 1.3, 2.1_

- [ ] 6. ファイル入力ハンドラー
- [ ] 6.1 (P) ドラッグ&ドロップでファイルを開く機能を実装する
  - Tauri の onDragDropEvent でファイルドロップを検知
  - ドラッグ中のドロップゾーンインジケーターを表示
  - ドロップ時に最初のファイルパスを取得しプレビュー表示に切り替え
  - 非 Markdown ファイルはプレーンテキストとして表示
  - _Requirements: 3.1, 3.2, 3.3_

- [ ] 6.2 (P) OS ファイル関連付けとファイルリンク起動を設定する
  - tauri.conf.json の bundle.fileAssociations で .md/.markdown を登録
  - Rust 側で RunEvent::Opened イベントを処理しファイルパスをフロントエンドに emit
  - フロントエンドで open-file イベントをリッスンしプレビュー表示に切り替え
  - _Requirements: 4.1, 4.2_

- [ ] 7. ウィンドウ管理とキーボード操作
- [ ] 7.1 キーボードショートカットでウィンドウを閉じる機能を実装する
  - Cmd+W キーイベントでウィンドウを閉じる
  - `:q` のキーシーケンスを検知してウィンドウを閉じる（vim 風操作）
  - キーシーケンスの状態管理（`:` 入力後に `q` を待つ）
  - _Requirements: 6.2_

- [ ] 7.2 単一インスタンスでのウィンドウ再利用を実装する
  - single-instance プラグインで既存インスタンスを検出
  - 新しいインスタンスの CLI 引数を既存ウィンドウに転送
  - 既存ウィンドウにフォーカスし、受け取ったファイルを表示
  - _Requirements: 6.3_

- [ ] 7.3 ウィンドウサイズと位置の永続化を設定する
  - window-state プラグインで自動保存・復元を有効化
  - capabilities に window-state プラグインの権限を追加
  - アプリ再起動時にウィンドウサイズ・位置が復元されることを確認
  - _Requirements: 6.4_

- [ ] 8. 大ファイル対応
- [ ] 8.1 大きな Markdown ファイルの段階的レンダリングを実装する
  - 1MB 以上のファイルを検知しチャンク分割処理に切り替え
  - 先頭部分を先にレンダリングし残りを非同期で追加
  - ローディングインジケーターを表示し進捗をユーザーに伝える
  - _Requirements: 2.3_

- [ ] 9. 結合テストと起動速度最適化
- [ ] 9.1 全フローの結合テストを実施する
  - CLI 起動 → ファイル読み取り → MD パース → プレビュー表示の一連のフロー
  - D&D → ファイル読み取り → プレビュー表示のフロー
  - ファイル不存在 → エラー表示のフロー
  - 非 MD ファイル → プレーンテキスト表示のフロー
  - 単一インスタンス再利用のフロー
  - _Requirements: 1.1, 1.4, 1.5, 3.1, 6.3_

- [ ] 9.2 起動速度を計測し 200ms 目標を達成する
  - performance.now() で WebView ロードからプレビュー表示までの時間を計測
  - ボトルネックの特定と最適化（Shiki 遅延ロード、バンドル分割等）
  - リリースビルドでの起動速度を計測し目標値を確認
  - _Requirements: 2.1_

## Parallel Execution Notes

タスク1（セットアップ）完了後、以下が並列実行可能:
- **Task 2.1 + 2.2**: Rust IPC コマンドは互いに独立
- **Task 4.1 + 4.2 + 4.3**: UI コンポーネントは互いに独立
- **Task 6.1 + 6.2**: ファイル入力ハンドラーは互いに独立

Task 3（MD パイプライン）は Task 2 に依存しないためセットアップ後すぐに開始可能。
Task 5（App シェル）は Task 2, 3, 4 の成果物を統合するため、これらの完了後に実行。
Task 7 は Task 5 完了後に実行（App シェルの状態管理に依存）。
Task 8, 9 は全機能実装後に実行。
